# 전처리 쉽게 이해하기 — 비전공자용 설명서

> 이 문서는 프로그래밍을 모르는 분도 **"우리 파이프라인이 도대체 뭘 하는 거냐"**
> 를 이해할 수 있도록 일상 비유로 풀어쓴 설명서입니다. 코드 한 줄도 안 나옵니다.

---

## 0. 우리가 하려는 일은?

LG 가전 신규 모델을 개발할 때, 기존 양산 모델과 비교해서 **어떤 부품이 어떻게
바뀌었는지** 한 줄씩 정리한 엑셀 파일을 사람이 손으로 채워 왔습니다. 그 파일을
"개발부품 마스터" 또는 "변경부품 리스트"라고 부릅니다.

문제는:

1. **양식이 4종**입니다. 옛날 양식 (20col), 중간 양식 (56col), 현재 운용 양식
   (96col), 미래 통합 양식 (v1.2). 사람마다, 시기마다, 모델마다 다른 양식을 씁니다.
2. **같은 의미인데 칸 이름이 다릅니다.** 어떤 양식은 `Base P/No`, 어떤 양식은
   `P/no.`, 어떤 양식은 `구분` 대신 `Classification`.
3. **같은 칸 안에 데이터 표기가 제각각입니다.** `Best-1`, `Best1`, `Best 1`,
   `B1` 다 같은 뜻인데 글자가 달라요.
4. **모든 파일이 합쳐져 있어야 검색·비교·집계가 됩니다.**

그래서 **"엑셀 잡탕을 깨끗한 표준 데이터로 만드는 작업"** = 전처리(preprocessing)
입니다. 마치 슈퍼마켓에 입고된 농산물을 (1) 종류별로 분류하고 (2) 흙 털고 (3)
규격에 안 맞는 건 따로 빼고 (4) 진열대에 올리는 과정과 비슷합니다.

---

## 1. 단계별로 — 슈퍼마켓 비유

### Step 0 — 인벤토리 (입고 점검)

> 트럭에서 박스를 내릴 때 **목록표**를 만드는 단계.

`data/raw/` 폴더에 엑셀 파일을 넣으면 우리가 **파일 이름, 시트 수, 각 시트의 행/열
개수, 파일 이름에 들어있는 모델 코드 힌트** 등을 자동으로 기록합니다.

이 목록만 봐도 "아 이 파일은 시트가 4개고 그 중 한 시트는 96열짜리니까 96col
양식인 것 같다" 정도의 1차 감을 잡습니다.

여기서 발견한 실제 문제: 그중 한 파일이 **엑셀 내부에 깨진 메타데이터** (`#N/A`)
때문에 일반적인 라이브러리로 열리지 않더라고요. 그래서 다른 라이브러리를 **자동
폴백**으로 붙여서 어떤 파일이든 일단 읽기는 되도록 해뒀습니다.

### Step 1 — 양식 분류 (어떤 종류의 농산물인가?)

> 사과/배/감 별로 박스를 나눠놓는 단계.

각 파일이 4가지 양식 중 어디에 속하는지를 **결정론적 규칙**으로 판별합니다. AI를
안 씁니다 — 사람이 정한 규칙만 씁니다. 예를 들어:

- 시트에 `History`라는 이름의 시트가 있고 열이 45~75개면 v1.2
- 열이 80개 넘고 `aaaa` 마커 셀이 있으면 96col
- 시트 이름이 `Better`/`Best`/`Good`로 시작하면 56col
- 단순한 단일 헤더에 20~35열이면 20col
- 시트 이름이 `ag-grid`고 헤더에 `Lvl`/`Check Out`이 보이면 PDM/PLM에서 뽑은
  BOM 트리 덤프 (bom_tree)

이 신호들을 각각 가중치로 합산해서 임계값(0.7)을 넘는 쪽으로 분류합니다. 두 양식
이 동시에 임계를 넘으면 **사람이 봐야 한다(needs_review)** 표시.

실데이터에서 발견한 것: 우리가 받은 5개 파일에 **5번째 양식**(ag-grid)이
끼어있었어요. 처음 계획엔 없던 종류라 신규로 분류기에 추가했습니다.

### Step 2 — 정답 양식 정의 (표준 진열대 규격)

> 슈퍼 진열대의 규격을 정하는 단계 — "사과는 이 크기 박스에 이런 라벨 붙여서 진열"

**현재 운용 중인 96col 양식을 우리의 답**으로 정했습니다. 어떤 양식이든 결국
96col 모양으로 변환해서 같은 통(데이터베이스)에 넣어요.

답에는 10개 고정 칸이 있습니다:
1. 변경 전 부품번호 (base_part_no)
2. 변경 후 부품번호 (new_part_no)
3. 부품명
4. BOM 레벨 (계층 깊이)
5. 부품 종류 (사출/Assy/전장/단품)
6. 변경 유형 (신규/변경/유지)
7. 변경점 (자유 텍스트)
8. 변경 사유 (자유 텍스트)
9. 수량
10. 모델 코드

그리고 **규칙(axiom)**도 함께 정해뒀습니다. 부품번호는 영문 2~3자 + 숫자
7~8자, 모델 코드는 영문 시작 + 영문/숫자 4~15자, 변경 유형은 신규/변경/유지 중
하나, 등급은 Best-1 / Better-1 / Good-1 8종 중 하나. **이 규칙에 안 맞는 값은
나중에 자동으로 격리됩니다.**

### Step 3 — 매핑 + 정규화 (실제 변환 작업)

> 박스 안의 농산물을 진열대 형태로 옮기는 단계 + 흙 털고 모양 다듬기.

세 가지 일이 동시에 일어납니다:

**① 매핑** = "이 양식의 어느 칸이 우리 답의 어느 칸과 같은 뜻인지" 옮기는 작업.
   `config/mapping_rules/96col.yaml`처럼 양식별로 룰 파일을 따로 두고 사람이
   직접 손으로 작성해서, 누가 봐도 이해되고 수정도 쉽게 했습니다.

   예: `"P/no."` 칸 → `base_part_no`. `"구분"` 또는 `"Classification"` 칸 → `change_type`.

**② Meta header 보강** = 실제 데이터를 보니, 모델 코드가 데이터 행에 안 들어
   있고 **시트 맨 위 메타 헤더(R2)**에만 한 번 적혀 있더라고요. 그래서
   `Base model | ... | LSIS6338F.ARSLSTC` 같은 라벨/값 쌍을 자동으로 읽어서 **모든
   데이터 행에 같은 모델 코드를 자동 채워 넣는** 보강 단계를 추가했습니다.

**③ 정규화** = 값 자체를 깨끗하게 다듬기. 예시:
   - `Best1` → `Best-1` (별칭 정리)
   - `내열\n보강` → `내열 보강` (줄바꿈 → 공백 + 압축)
   - 전각 영문/숫자 → 반각 (`ＡＢ1234567` → `AB1234567`)
   - 한글 NFD ↔ NFC 통일
   - `N/A` / 빈 문자열 / 공백 → null
   - `0987654321` 같은 leading-zero 부품번호 보존 (Excel이 함부로 정수로 잘라먹지
     않도록)
   - **10가지 함정**을 모두 테스트로 잡아두고 있어요.

### Step 4 — 검증 + Quarantine (불량품 솎아내기)

> 진열대 올리기 전에 **체크리스트로 점검**해서, 규격 미달은 따로 보류함에 넣는 단계.

전체 결과를 받으면 **15개 지표**로 측정합니다:

- **A. 통일성**: 필수 컬럼 다 들어왔나? 자료형 맞나? 값이 패턴에 맞나? 모델 코드
  참조 무결한가? 원본 행 보존되었나?
- **B. 품질**: 필수 필드의 결측 비율은? 중복 행은? 규칙 위반 비율은? 이상치는?
- **C. 처리 통계**: 입력 행 수, 출력 행 수, 격리 행 수, 드롭 행 수, 드롭 사유별 카운트.

각 지표마다 **합격 임계**가 있고, 하나라도 미달이면 **NOT ACCEPTABLE** 판정.

**Quarantine = 격리함, 쓰레기통이 아닙니다.** 실패한 행은 버리지 않고 따로
`data/quarantine/<run_id>/` 폴더에 통째로 저장해서 **사람이 검토해서 살릴 수
있게** 해놓습니다. 예: 필수 필드가 비어있어서 격리된 행 → 사람이 다른 자료 보고
채워서 다시 처리.

**Golden diff** = 사용자가 직접 손으로 만든 결과(`data/golden/`)와 자동
결과를 매 처리마다 자동으로 비교합니다. 자동화가 어디서 사람과 다른 판단을 내렸는지
보여줘요. 일치율 95%+ 가면 자동화 신뢰. 지금은 사용자 손작업 결과가 없어서 이 비교는
대기 중.

**리포트**: `data/reports/<run_id>.md` 파일에 모든 결과가 markdown으로 정리됩니다.
표 + 판정 + 다음 명령 안내 포함.

### Step 5 — DB 적재 (진열대에 올리기)

> 검수 통과한 농산물만 매대에 진열.

검증을 통과한 batch만 PostgreSQL 데이터베이스에 적재합니다. 이때 5개 테이블에
들어갑니다:

- `parts`: 부품 마스터
- `models`: 모델 마스터
- `bom_edges`: BOM 트리 (부모-자식 관계)
- `change_events`: 변경 이벤트 한 건
- `event_details`: 변경 이벤트에 딸린 부가 정보 (JSON)

거기에 **embedding** 컬럼이 따로 있어서 `change_point` 자유 텍스트를 숫자
벡터로 변환해 저장합니다. 이 벡터로 "내열 보강과 비슷한 과거 사례 찾아줘" 같은
**의미 검색**이 됩니다.

## 2. 안전장치 — Dry-run / Commit / Rollback

> "확정 전에 미리보기, 잘못되면 되돌리기" 같은 거.

처음부터 완벽한 룰은 불가능합니다. 그래서:

1. **Dry-run** = 일단 처리해서 임시 폴더에 저장. DB에는 아직 안 들어감. 리포트
   생성. 사람이 검토.
2. **Commit** = 검증 통과한 dry-run을 진짜로 DB에 적재. `run_id`로 어떤 배치
   인지 추적 가능.
3. **Rollback** = 적재 후 문제 발견되면 그 `run_id`만 정확히 삭제. **Ctrl+Z**
   같은 거.

검증 게이트(15지표)가 통과 못 하면 **commit이 거부**됩니다. 그래서 망가진 데이터가
DB에 들어가는 일은 자동으로 차단됩니다.

## 3. 자동화의 학습 곡선 (현실)

처음 실데이터를 돌렸을 때:
- **1차**: 0/721 = 0% 통과 (모든 행 quarantine)
- **2차** (메타 헤더 model_code 자동 추출 보강 후): 502/713 = 70%
- **3차** (model_code 패턴 완화 + 96col part_type 선택 처리): **513/713 = 72%**

이런 식으로 **데이터를 한 번 보고, 룰을 다듬고, 다시 돌리는 과정을 반복**합니다.
"완벽한 자동화"가 아니라 **"점진적으로 사람 손이 덜 가게 만드는 과정"**.

목표는:
- 1~2주차: 자동화 정확도 50~70%, 나머지는 사람이 보강
- 3~4주차: 80~90%, 사람은 needs_review만 검토
- 안정화 후: 사람은 **새 양식 등장** 같은 예외만 처리

## 4. 결정론적 vs LLM

> "AI 안 쓰는 거 맞나요?" 라는 질문에 대한 답.

이 파이프라인의 **모든 변환 코드는 100% 결정론적**입니다. 같은 입력 → 항상 같은
출력. LLM(GPT 같은 거)이 끼는 곳은 **없습니다**.

그럼 LLM은 어디 쓰냐고요?

- **현재(MVP)**: 한 군데도 안 씁니다.
- **다음 단계 (Phase 2)**: 사용자가 자연어로 "내열 220도에서 240도로 올린
  사례 찾아줘"라고 물으면 → LLM이 검색 의도를 파악해서 DB 쿼리를 만들고 →
  의미 검색(embedding)으로 비슷한 과거 사례 찾고 → 결과 요약. 사용자 손작업
  보조용입니다.
- **변환 자체에는 절대 안 씁니다.** 변환은 검증 가능하고 재현 가능해야 하니까요.

---

## 5. 우리가 만들지 않은 것 (해명)

- **새로운 ML 모델 학습**: 안 합니다. 룰만 정교화.
- **풀 자동화**: 100% 자동화는 목표 아닙니다. 사람 검토 부담을 줄이는 게 목표.
- **모든 양식의 완전 지원**: ag-grid(BOM 트리 덤프)는 지금은 "별도 영역
  (`bom_edges` 테이블)" 으로 인식만 하고 적재는 다음 작업.

## 6. 한 페이지 요약

| 단계 | 비유 | 산출물 |
|---|---|---|
| Step 0 | 입고 점검 | `data/interim/file_inventory.parquet` |
| Step 1 | 종류별 분류 | 양식 분류 (4종 + bom_tree) |
| Step 2 | 표준 진열대 규격 | 96col 정답 스키마 + axiom 규칙 |
| Step 3 | 매핑 + 흙 털기 | 정규화된 데이터프레임 |
| Step 4 | 체크리스트 검수 + 보류함 | 15지표 리포트 + quarantine |
| Step 5 | 진열대 적재 | PostgreSQL 5 테이블 |

**전체 명령**:

```bash
# 1. 환경 + 의존성
uv sync --extra dev --extra viz

# 2. 인벤토리
uv run python -m src.cli inventory

# 3. 분류
uv run python -m src.cli classify data/raw --all

# 4. 전체 dry-run + 리포트
uv run python -m src.cli pipeline run

# 5. 리포트 검토 후 commit
uv run python -m src.cli pipeline commit --run-id <ID>

# 6. DB 적재 (Postgres 떠 있을 때)
uv run python -m src.cli db load --run-id <ID>

# 7. 시각화 한 번에 다시 만들기
uv run python scripts/build_walkthrough.py
```

자세한 단계별 시각화는 [`walkthrough.md`](walkthrough.md) 참조.
